# 编码规范
## 模型响应风格
- 使用中文思考及响应
## 方案设计
- 如果是方案设计涉及阶段,先分析方案实现的优劣,并给于方案补充建议.
## 程序实现
- 引用组件或库时,优先引用最新稳定版
- 方法需要写明注释及doc
- 方法需要有对应的测试代码
- 代码完成后需要编译并运行通过
- 代码优先使用性能高的实现方式
## Flutter 开发规范
- **UI 与逻辑分离**：Widget 仅负责 UI 展示，业务逻辑应抽离至 State Management 层（如 Provider/Bloc/GetX）。
- **组件拆分**：避免过大的 build 方法，将复杂 UI 拆分为独立的 StatelessWidget/StatefulWidget。
- **异步处理**：避免在 build 方法中执行耗时操作或副作用，使用 FutureBuilder/StreamBuilder 或状态管理处理异步数据。
- **资源管理**：图片、字体等资源路径应统一管理，避免硬编码。
- **性能优化**：合理使用 `const` 构造函数，避免不必要的 Widget 重建。
- **代码风格**：遵循官方 lints 规范，保持代码整洁一致。
## 异常处理
- 优先使用模型尝试解决,如果尝试超过3种方案仍未解决,分析原因并提醒开发者介入.
## 通用规范
- 避免不必要的对象复制或克隆
- 避免多层嵌套，提前返回
- 使用适当的并发控制机制
- 代码分层,模块分离,职责清晰
- 避免代码重复,提取公共逻辑

## 代码可读性
### 命名约定
- 使用有意义的、描述性的名称
- 遵循项目或语言的命名规范
避免缩写和单字母变量（除非是约定俗成的，如循环中的 `i`）

### 代码组织
- 相关代码放在一起
- 一个函数只做一件事
- 保持适当的抽象层次

### 注释与文档
- 注释应该解释为什么，而不是做什么
- 为公共 API 提供清晰的文档
- 更新注释以反映代码变化
